# ============================================================
# CORE ENGINEERING RULES — 高内聚低耦合工程规范
# 适用范围: 全项目 / Always Active
# ============================================================

**权衡说明**：本规范整体偏向谨慎而非速度。对于极简单的任务，可酌情判断执行力度。

## 〇、黄金法则（优先级最高，任何情况不得违反）

1. **禁止打补丁思维**：发现问题时，先定位根因，再从源头修复；严禁在现有逻辑外层堆叠 if/else、try/catch 或临时变量来绕过问题。
2. **改动必有理由**：每次代码变更必须能用一句话说清"为什么改"，无法说清则重新分析。
3. **文档与代码同步**：任何影响接口、行为、配置或依赖的变更，必须在同一次提交中同步更新相关文档（见第五节）。
4. **宁可不写，不可乱写**：不确定的逻辑不要用注释掩盖，要明确标记 `TODO(reason):` 并说明原因。

## 一、架构设计原则

### 1.1 高内聚低耦合（核心）
- 每个模块/类/函数只负责**一件事**（单一职责原则 SRP）。
- 模块间通信通过**明确定义的接口/契约**，禁止跨层直接访问内部实现。
- 依赖方向必须单向，例如：`UI → 业务逻辑 → 数据层`，不得反向依赖。
- 共享逻辑抽取到独立工具模块，禁止在多处复制粘贴同一段代码（DRY原则）。

### 1.2 SOLID 原则执行清单
- **S** — 一个函数不超过 40 行；一个类不超过 200 行；超出时需评估是否违反 SRP；若职责单一但逻辑较长，允许例外但须在代码中注释说明原因。
- **O** — 新增功能通过扩展（新文件/新类）实现，不修改已稳定运行的模块。
- **L** — 子类/实现类可替换父类/接口，不得收窄行为。
- **I** — 接口粒度要小，禁止一个接口承担多种不相关职责。
- **D** — 依赖抽象而非具体实现；核心业务逻辑不得直接 import 第三方库（通过适配器隔离）。

### 1.3 简洁优先

**最小化代码**：用最少代码解决问题，不做多余拓展。

- 仅实现需求明确要求的功能，可以给用户建议但不额外新增功能。
- 仅使用一次的代码，不做抽象封装。两次以上则遵循DRY原则。
- 未被要求的"灵活性"或"可配置性"不要加。
- 针对不可能发生的场景不编写异常处理逻辑或边界检查。
- 写了 200 行但 50 行能解决的，务必重构精简。

**自我审视**：「一个资深工程师看到这段代码会认为这段代码过于繁琐吗？」如果答案是"会"，简化它。

### 1.4 分层结构规范（参考架构，根据项目类型调整）
```
src/
  core/          # 纯业务逻辑，零外部依赖
  services/      # 用例/应用层，协调 core 与外部
  adapters/      # 第三方库/API 适配器（隔离外部变化）
  utils/         # 纯函数工具，无副作用
  types/         # 全局类型/接口定义
  tests/         # 测试文件，镜像 src 目录结构
  docs/          # 文档（见第五节）
```

### 1.5 依赖管理
- 引入新的第三方依赖前，必须评估：维护活跃度、License 兼容性、包大小/性能影响。
- 禁止为解决单一小问题引入整个重型库（能用标准库实现的不引入外部包）。
- 锁定依赖版本（`package-lock.json` / `poetry.lock` 等），禁止使用浮动版本（`^` / `~` 酌情使用）。
- 定期（每季度）审查并更新过期依赖，不得让依赖停留在有已知漏洞的版本。

## 二、编码规范

### 2.1 命名
- 变量/函数：`camelCase`（JS/TS）或 `snake_case`（Python），必须是**动词+名词**描述
  行为，如 `fetchUserProfile()` / `fetch_user_profile()`，禁止 `data`、`temp`、`info`
  等模糊命名。
- 类/类型/接口：`PascalCase`。
- 常量：`UPPER_SNAKE_CASE`，禁止魔法数字，全部提取为命名常量。
- 文件名：JS/TS 组件用 `kebab-case`，Python 模块用 `snake_case`；须与其导出/定义的
  主要内容同名。
- **目录命名**：一律使用 `snake_case`的格式，禁止使用 `-` 作为连接符；
  例如 `agent_test_tool`，而非 `agent-test-tool`。

### 2.2 函数设计
- 函数参数不超过 **3个**；超出时用对象参数（Options Object 模式）。
- 每个函数的**返回语义必须单一**，成功/失败通过类型区分，不混用）。
- 纯函数优先：无副作用的逻辑必须写成纯函数，便于测试。
- 异步函数统一使用语言推荐的同步风格写法（JS/TS 用 async/await，禁止混用
  .then() 链式调用；Python 用 async/await；其他语言遵循各自社区最佳实践），
  保持风格统一，禁止在同一项目中混用多种异步模式。

### 2.3 禁止行为清单
- ❌ 禁止 `any` 类型（TypeScript项目），必须显式定义类型。
- ❌ 禁止空 `catch` 块（`catch(e){}`），必须记录错误或重新抛出。
- ❌ 禁止在业务逻辑中直接使用 `console.log`，使用统一 logger。
- ❌ 禁止全局变量（除配置常量外）。
- ❌ 禁止在一个文件中定义超过 **1个** 主要类/模块。
- ❌ 禁止注释掉旧代码后保留，删除即可（有版本控制兜底）。

## 三、防退化机制（针对 Vibe Coding 场景）

### 3.1 编码前必须先思考（Think Before Coding）
**不主观臆断，不掩饰疑问，主动说明方案取舍**
正式编写代码前：
- **清晰列明所有预设条件，存有疑问及时沟通。**——实施前明确说出你的全部假设。若对需求、设计或代码行为有任何不确定，必须主动提问，不得自行猜测。
- **如果需求存在多种解读方式，全部列出**——存在多种可行的解释或实现方案时，将它们全部列出并说明权衡，然后等待决策，而非自行挑选一种。
- **如果有更简洁的实现思路，主动提出；必要时提出异议。**——若发现明显更简单的实现方式，必须明确指出来。当建议不被采纳时，按要求执行；当需求本身不合理时，应当提出异议。
- **如果某处内容理解不清晰，停下来**——遇到任何不理解的地方，立刻停下。明确说出“哪里不清楚、为什么不清楚”，并询问确认。

### 3.2 修复前必做的分析
在修改任何 bug 之前，**必须先回答**：
1. 根因是什么？（不是表象）
2. 这个修复会影响哪些其他模块？
3. 是否有更干净的实现方式，而不是在现有代码上打补丁？

若无法回答，停止修改，重新阅读相关模块代码。

### 3.3 目标驱动执行（Goal-Driven Execution）
**将任务转化为可验证的目标，循环执行直到验证通过。**

示例转化：
- "增加校验逻辑" → "先编写非法输入的测试用例，再修复代码使用例通过"
- "修复 bug" → "先编写可复现问题的测试用例，再修复代码使用例通过"
- "重构代码 X" → "确保重构前后所有测试用例均可正常运行通过"

对于多步骤任务，先输出简要执行计划：
```
1. [步骤] → 校验项：[检查内容]
2. [步骤] → 校验项：[检查内容]
3. [步骤] → 校验项：[检查内容]
```
清晰的验收标准能让你独立完成迭代；若标准模糊（如仅要求 “实现功能”），则需要反复沟通确认。

### 3.4 重构触发条件（强制）
以下情况**必须重构**，不得继续堆积：
- 同一个函数被修改超过 **3次** 仍有问题 → 说明设计有误，需重新设计。
- 一个文件引用超过 **5个** 不同模块 → 违反低耦合，需拆分职责。
- 修复一个 bug 引入了新的 bug → 说明存在隐性耦合，需解耦后重测。
- 代码中出现超过 **2处**相同逻辑块 → 立即抽取为公共函数。

### 3.5 变更范围控制
**只改动必要内容，仅清理自身产生的冗余代码**

修改已有代码时：
- 每次 AI 辅助修改**只针对一个明确目标**，不要擅自优化周边代码、注释或调整代码格式。
- 不对运行正常的代码做重构。
- 沿用项目现有代码风格，即便个人写法不同也要保持一致。
- 发现无关的废弃代码，及时向用户发出提醒，并注释添加"TODO(cleanup): 此处为废弃逻辑，待确认后删除"标记，不要擅自删除。
- 改动超过 **3个文件** 时，必须先输出变更影响分析，再执行。

当你的改动产生了无用代码：
- 删掉因本次修改而不再使用的导入项、变量和函数。
- 非经允许，不得删除原本就存在的废弃代码。

**检验标准**：每一行修改的代码，都必须与用户需求直接对应。

## 四、测试规范

### 4.1 测试策略
- 新增功能必须附带**单元测试**，覆盖正常路径 + 至少2个边界情况。
- 修复 bug 时，必须先写**能复现该 bug 的测试**，再修复，确保测试由红变绿。
- 测试文件与被测文件同名，按语言惯例命名后缀（JS/TS：.test.ts / .spec.ts；
  Python：test_<name>.py；Go：<name>_test.go），镜像放置于 tests/ 目录。
- 涉及外部系统（数据库、第三方 API、消息队列）的模块，必须附带**集成测试**，
  使用 mock 或测试专用实例隔离外部依赖。
- 核心业务流程（关键用户路径）必须有**端到端测试（E2E）**覆盖，
  确保跨模块协作正确。

### 4.2 测试命名规范
测试函数命名格式：test<预期行为>when<条件>（各语言按惯例实现），测试描述必须是行为陈述，不得用无意义命名。
例：testReturnEmptyListWhenUserHasNoOrders() / test_return_empty_list_when_user_has_no_orders()

### 4.3 测试文档同步
- 每次新增/删除测试用例，同步更新 `docs/TESTING.md`（记录测试覆盖范围和测试策略说明）。

## 五、文档同步规范（强制）

> 以下文档必须与代码变更**同一次提交**更新，不得分离。

| 变更类型 | 必须更新的文档 |
|---|---|
| 新增功能 / API | `README.md`（功能说明）+ `docs/API.md` |
| 修改函数签名 / 接口 | `docs/API.md` + 相关模块的内联文档注释（JSDoc / docstring / Javadoc，按语言惯例） |
| 修改配置项 | `README.md`（配置说明章节）+ `docs/CONFIGURATION.md` |
| 修改数据结构/数据库 | `docs/DATA_MODEL.md` |
| 修复 Bug | `CHANGELOG.md`（按日期分组，记录修复内容和根因） |
| 新增/修改环境依赖 | `README.md`（安装章节）+ `docs/SETUP.md` |
| 新增/修改测试 | `docs/TESTING.md` |
| 架构调整 | `docs/ARCHITECTURE.md`（含模块图） |

### README.md 最小结构要求
```markdown
# 项目名称
## 项目简介（一句话）
## 快速开始
## 功能列表（与当前代码保持同步）
## 配置说明
## 项目结构
## 测试
## 变更日志（链接到 CHANGELOG.md）
```

## 六、错误处理规范

- 错误分三层处理：**捕获层**（adapter）→ **转换层**（service，转为业务错误）→ **展示层**（UI/API响应）。
- 业务错误使用自定义 Error 类，携带 `code`、`message`、`context` 三个字段。
- 所有错误必须通过统一 logger 记录，格式：`[模块名][错误级别] 描述 | context: {...}`。
- 用户可见的错误信息必须友好，技术细节不得暴露给前端。

## 七、安全与性能基线

- 所有外部输入（API、用户表单、环境变量）必须在进入业务逻辑前**验证和sanitize**。
- 敏感信息（密钥、Token、密码）禁止出现在代码、日志、注释中，使用环境变量管理。
- 环境配置文件分层管理：`.env.example`（提交到仓库，含所有 key 但无值）+
  `.env.local`（本地实际值，加入 .gitignore）。
- 禁止在代码中硬编码任何环境相关值（端口、域名、超时时间），统一从配置读取。
- 配置项变更必须同步更新 `.env.example` 和 `docs/CONFIGURATION.md`。
- 数据库查询禁止拼接字符串，使用参数化查询/ORM。
- 性能敏感路径（循环、递归、I/O）需在行内或上方标注 PERF: 注释说明设计考量
  （格式按语言注释风格：Python 用 # PERF:，JS/TS/Go 用 // PERF:）。
- 避免在循环内部进行数据库查询或网络请求（N+1问题）。

## 八、Git 提交规范

提交信息格式（Conventional Commits）：
```
<type>(<scope>): <简短描述>

[可选正文：解释 why，不是 what]
[可选：影响的文档列表]
```
type 枚举：`feat` | `fix` | `refactor` | `docs` | `test` | `chore` | `perf`

- `fix` 类型的提交**必须**在正文中说明根因。
- `refactor` 类型的提交**必须**说明重构前的问题和重构后的改善点。
- 单次提交只做一件事，禁止"fix bugs and add features"混合提交。

## 九、AI 辅助开发行为约束

- **先理解，后动手**：在修改代码前，必须先阅读并理解被修改模块的完整上下文。
- **最小化变更**：只修改达成目标所必须的代码，不"顺手"优化其他地方。
- **不造轮子**：项目中已有的工具函数/组件，优先复用，不重新实现。
- **不假设**：不确定某个函数的行为时，先查看其实现和测试，不猜测。
- **变更透明**：每次修改后，输出一个简短的影响分析：「修改了什么 → 影响了哪些模块 → 需要同步更新哪些文档」。
