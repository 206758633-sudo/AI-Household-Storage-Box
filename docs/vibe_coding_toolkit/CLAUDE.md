## 工作规范

### 1. 先讲逻辑再动手
任何超过 10 行、或涉及 2 个以上文件的改动,动代码前先用文字说明:
- 打算改什么
- 影响哪些文件
- 为什么这么改

等用户确认后再写代码。方向错了能立刻调头,不会写一堆才推翻。

### 2. 改动分级 A / B / C

**判断不准时,向高一级靠。** 多花一分钟说方案,远比搞砸后回退便宜。

**A 级 — 出错回不去 / 碰核心资产**(任一即是 A)
- 改数据库结构、修改或删除已有数据
- 改认证、权限、密钥、加密相关
- 改支付、计费、订阅相关
- 调外部付费 API、发送邮件 / 短信 / 推送
- 不可逆 git 操作:`force push` / `reset --hard` / 删分支
- 改 CI/CD、部署脚本、生产环境配置
- 跨 3 个以上文件的架构性重构

动作:
1. 先写完整方案(目标、改动点、风险、回退方式),等用户**明确说"可以"**
2. 必须开新分支,不在主分支直接动
3. 改完不直接合,等用户审完

**B 级 — 用户能感知但可回滚**
- 用户能看到的功能 / UI / 文案
- 新增功能(新文件、新接口,不动现有)
- 单个模块内业务逻辑变更
- 改对内 API 接口签名

动作:动手前一句话说意图,做完跑测试与 lint,单独 commit。

**C 级 — 纯局部 / 可逆 / 不影响运行**
- 注释、文档、README
- 单文件内格式化、变量改名(不影响外部调用)
- 加日志、调日志级别
- 加单元测试
- 修 typo

动作:直接做,与其他改动一起 commit。

### 3. 即时 git,保证能回退
- 每完成一个能跑起来的小步骤,就 `git commit` 一次
- A 级改动必须先 `git checkout -b` 开新分支
- commit message 格式:`类型: 一句话说明`(类型:feat / fix / refactor / docs / test / chore)
- 用户问"现在能跑吗?"——若不能,说明上一次该提交而没提交

### 4. 保持代码健康
- 写之前先搜:项目里有没有类似功能可以复用,有则用
- 单个函数尽量 < 50 行,单个文件尽量 < 300 行;超过先想能否拆分
- 同样的逻辑出现 2 次可以容忍,第 3 次出现前必须抽出来
- 写完顺手清理:调试 print、注释掉的废代码、临时变量
- 留 TODO 必须带:`// TODO: [谁] [做什么] [为什么暂时不做]`

### 5. 软件工程规范
- **单一职责**:一个函数 / 模块只做一件事
- **命名清晰**:看名字就知道在干嘛;宁可长,别缩写到看不懂
- **就近原则**:相关代码放一起,不按"类型"散到各目录
- **不过度设计**:不为"未来可能用到"提前抽象;等真用到第二次,再抽

### 6. API 与密钥安全
- **绝不**把 API key / token / 密码 / 私有 URL 写进代码
- **绝不**把 `.env` / `secrets.*` 加入 git (`.gitignore` 必须包含)
- 每次 commit 前自查 `git diff`,扫一眼有无误粘密钥
- 第三方 API 调用前,确认费用与频率限制,在方案里说明
- 用户数据(手机号、邮箱、地址、身份证)不写进日志

### 7. 收尾动作
每完成一段改动,依次:
1. 跑测试(若有)
2. 跑 lint / typecheck(若有)
3. 自检是否符合上面的分级要求
4. `git commit`,说清改了什么、为什么

任何一步失败,先修好再继续,不带红灯前进。

### 8. 依赖纪律
- 加新依赖前必须先说:"打算用 `[库名]` 解决 `[什么问题]`,有无现成替代?"
- 用户同意后再装,装完更新本文件"技术栈"段
- 不装来路不明、长期未维护、下载量极低的库
- 不为一个小功能装一个大库

### 9. 不擅自删除、不擅自大重构
- 看到不理解的代码,先**问**为什么这么写,不直接删
- 大重构(3+ 文件、改函数签名)属 A 级,按 A 级走
- "顺手优化"超过 5 行,先说明再动

### 10. 不确定就停下来问
- 需求模糊、命名纠结、库选型不定——**问**,不要猜
- 提供 2-3 个具体选项让用户选,比"你想怎么做?"高效
- 问完先停,不要边等回复边继续做下一步

### 11. 自我维护本文件
- 本文件目标 ≤ 150 行,超了触发剪枝
- `#` 键沉淀仅用于**长期规则**,不沉淀临时琐事
- 每月或大版本回看一次,删过期内容
- 上半"项目事实"变化时同步更新
